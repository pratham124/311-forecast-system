# Architecture Assumptions

## UC-01 Scheduled 311 Data Pull

- The configured schedule may be simulated by a manual trigger in test environments, as long as the same ingestion flow and observable run status are exercised.
- The exact schedule interval is outside UC-01 scope; UC-01 only requires that a configured schedule exists and can trigger the ingestion run.
- The source request uses an exclusive last-successful-pull cursor: each successful run stores a cursor value from that run, and the next run requests only records newer than that stored cursor.
- If no last-successful-pull cursor exists yet, the first run requests the full available source dataset and establishes the first cursor only after a successful validated store.
- Outcome rules are fixed as follows:
  - New records success: create a new stored `DatasetVersion` and mark it current after validation and storage both succeed.
  - No-new-records success: create no new stored `DatasetVersion`; keep the current dataset unchanged.
  - Authentication failure: create no new `DatasetVersion`; keep the current dataset unchanged.
  - Timeout or source unavailable: create no new `DatasetVersion`; keep the current dataset unchanged.
  - Validation failure: allow only a transient candidate dataset for validation; create no new stored `DatasetVersion`; keep the current dataset unchanged.
  - Storage failure: do not treat the candidate dataset as a stored `DatasetVersion`; keep the current dataset unchanged.
- Dataset state terms are fixed as follows:
  - Candidate dataset: data retrieved for the current run and under validation or pre-storage handling; it is never current.
  - Stored `DatasetVersion`: a validated dataset that has been durably stored; it may be current or inactive.
  - Current dataset: the stored `DatasetVersion` referenced by the current-dataset marker for downstream consumers.
- No partial activation means the current-dataset marker can change only after validation succeeds and durable storage succeeds for the same dataset.
- "Current dataset is queryable" means acceptance checks can retrieve at least the source identifier, current dataset version identifier, activation timestamp, activating run identifier, and record count.
- A `FailureNotificationRecord` is recorded in a queryable monitoring record store and contains at minimum the failed run identifier or run timestamp, failure reason category, failed status, recorded-at timestamp, and a human-readable failure summary.
- UC-01 scope is limited to scheduled ingestion, validation, storage, current-dataset activation, and failure monitoring records.
- UC-01 does not add forecasting behavior, dashboard UI behavior, or email alert delivery unless a later use case explicitly requires them.
- Dataset version identifiers may be generated by the system, provided the active dataset can be queried after each run.
