# Feature Specification: Configure Alert Thresholds

**Feature Branch**: `013-uc-13-configure-alert-thresholds`
**Status**: Completed (Aligned with Implementation)
**Last Updated**: 2026-04-10

## Overview

UC-13 provides an authenticated interface for operational managers to configure alerting limits (thresholds) for service categories. In this simplified version, all alerts are delivered exclusively to the system's internal **Dashboard**, and thresholds are managed at the **Service Category** level only.

## User Scenarios

### User Story 1 - Manage alert thresholds (Priority: P1)

As an operational manager, I want to set and update alert thresholds for specific service categories so that I am notified via the dashboard when forecasted demand exceeds manageable limits.

**Acceptance Scenarios**:
1. **Given** an operational manager opens the Alert Review page, **When** they view the configuration section, **Then** they see a list of active thresholds by service category.
2. **Given** a service category exists in the system, **When** the manager adds a new threshold value, **Then** the system saves the configuration and immediately triggers an evaluation for that category.
3. **Given** an existing threshold, **When** the manager edits or deletes it, **Then** the system updates the active set and reflects the change in future alert cycles.

## Requirements

### Functional Requirements
- **FR-001**: The system MUST allow authenticated operational managers to manage thresholds.
- **FR-002**: Thresholds MUST be configurable by service category.
- **FR-033**: Every threshold MUST default to the "dashboard" notification channel.
- **FR-004**: Adding or updating a threshold MUST trigger a background re-evaluation of current forecasts for that category.
- **FR-005**: The system MUST support standard CRUD operations (List, Create, Update, Delete) for active thresholds.

### Simplified Scope (Out of Scope)
- **Geography-specific thresholds**: All thresholds apply globally to the category.
- **External channels**: Email, Slack, SMS, etc. are currently disabled in favor of dashboard-only delivery.
- **Immutable Versioning**: Thresholds are mutable and updated in-place (with active/inactive status).

## Success Criteria

- **SC-001**: Operational managers can successfully save a threshold and see it immediately appear in the active list.
- **SC-002**: Alerts generated by background evaluations appear in the "Recorded Alerts" list with the correct category and threshold value correlation.
- **SC-003**: Inactive thresholds do not trigger new alerts.

## Clarifications

- **Channels**: The notification channel is always `["dashboard"]`. This is hardcoded in the frontend and handled by a mock delivery client in the backend that always succeeds.
- **Triggers**: Evaluations are triggered automatically on threshold change and on new forecast publication.
